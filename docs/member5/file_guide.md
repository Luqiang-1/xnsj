# 成员5工作文件说明

本说明用于向老师展示成员5在“数据采集、清洗与知识库入库工具”方面完成的工作。成员5主要负责资料来源登记、采集过程演示、网页噪声清洗、Markdown 初稿生成、正式知识库整理记录补充，以及新增资料入库流程说明。

## 一、正式知识库文件夹：knowledge/

`knowledge/` 是系统真正参与问答检索的本地知识库目录。系统启动后，`rag_engine.py` 会读取该目录下的 Markdown 文件，并把其中的小节切分成知识片段用于问答。

成员5对该目录做的工作：

* 保留原有作物资料作为正式问答数据来源。
* 给全部 73 个作物 Markdown 文件补充了整理记录。
* 每个文件末尾统一增加：

```markdown
## 整理记录

* 整理人员：Kevin
* 整理时间：2026-05-19
```

这样做的作用是让知识库资料有明确的维护人和整理时间，方便答辩时说明资料不是无来源、无管理的散乱文本。

## 二、成员5演示工具文件夹：tools/

### tools/member5_crawler.py

这是成员5编写的资料采集与清洗演示工具。该脚本可以运行，主要用于展示“如果需要批量整理公开农业资料，成员5如何完成采集、清洗、转 Markdown 初稿”的流程。

脚本功能包括：

* 读取 `data/member5/source_register.csv` 来源登记表。
* 支持读取离线 HTML 文件，也预留了读取公开 URL 的能力。
* 提取 HTML 正文内容，跳过 `script`、`style`、页眉、页脚、导航等无关区域。
* 清理网页噪声，例如首页、导航、版权声明、上一篇、下一篇、联系方式等。
* 删除重复段落和无句末标点的短标题行。
* 按关键词把资料初步归类到统一知识库小节。
* 生成 Markdown 采集初稿。
* 输出 JSONL 格式运行日志，记录每次采集和清洗统计。

演示命令：

```powershell
python tools/member5_crawler.py --sources data/member5/source_register.csv
```

如果本机 `python` 命令不可用，可以使用实际 Python 路径运行。

## 三、成员5数据痕迹文件夹：data/member5/

`data/member5/` 用于保存成员5的采集、清洗和入库过程材料。这个目录不直接参与系统问答，正式问答仍然读取 `knowledge/`。

### data/member5/source_register.csv

来源登记表。用于记录每条资料的来源信息。

主要字段包括：

* `crop`：作物名称，例如草莓、葡萄。
* `category`：作物分类，例如浆果类。
* `source_type`：来源类型，例如离线 HTML 文件。
* `source`：来源文件路径。
* `source_site`：来源网站或来源说明。
* `source_url`：原始来源链接。
* `collector`：采集人员，这里为 Kevin。
* `collect_date`：采集日期。
* `remark`：备注说明。

这个文件用于体现成员5在资料采集前做了来源登记，而不是直接把内容随意复制进知识库。

### data/member5/sample_pages/

离线演示网页目录。里面保存了两个示例 HTML 页面：

* `strawberry_gray_mold.html`：草莓灰霉病与设施管理示例资料。
* `grape_management.html`：葡萄设施栽培与采收管理示例资料。

使用离线页面的原因是课堂展示时不依赖网络，避免真实网页打不开、编码异常或访问受限。它们用于演示爬虫工具如何从网页结构中提取正文并清理噪声。

### data/member5/cleaning_rules.md

数据清洗规则说明。该文件总结了成员5清洗资料时使用的规则，包括：

* 删除网页导航、页眉、页脚、版权声明、联系方式等。
* 删除“当前位置、上一篇、下一篇、相关阅读、打印、分享”等页面功能文字。
* 删除重复段落。
* 删除过短、无上下文、无法作为问答依据的碎片。
* 删除无句末标点的短标题行。
* 保留来源网站、来源链接、采集人员和采集时间。

这个文件用于说明数据清洗不是随意处理，而是有统一规则。

### data/member5/drafts/

采集初稿输出目录。运行 `tools/member5_crawler.py` 后会自动生成。

当前示例包括：

* `data/member5/drafts/浆果类/草莓_采集初稿.md`
* `data/member5/drafts/浆果类/葡萄_采集初稿.md`

这些文件是脚本根据示例网页生成的 Markdown 初稿，结构与正式知识库一致，包括：

* 品种介绍
* 种植方式
* 环境需求
* 常见病虫害与解决方案
* 农药使用
* 采收与销售
* 待人工校对
* 资料来源
* 整理记录

注意：`drafts/` 下的文件只是采集初稿，不直接进入正式问答系统。需要成员2人工校对后，才能整理到 `knowledge/`。

### data/member5/logs/crawl_log.jsonl

采集运行日志。运行脚本后自动生成，每一行是一条 JSON 记录。

日志内容包括：

* 运行时间
* 作物名称
* 来源类型
* 来源路径
* 采集人员
* 生成目标文件
* 清洗统计信息

清洗统计示例字段：

* `raw_lines`：原始行数。
* `kept_lines`：保留行数。
* `duplicate_lines`：删除的重复行数。
* `noise_lines`：删除的噪声行数。
* `short_lines`：删除的过短行数。
* `heading_lines`：删除的短标题行数。

这个日志用于展示成员5的工作过程可追踪、可复查。

## 六、答辩时可以这样介绍

成员5的工作不是直接负责问答算法，而是负责知识库资料进入系统前的准备工作。具体包括：

1. 先登记资料来源，保证资料可追溯。
2. 用采集工具读取网页或本地 HTML 页面。
3. 清洗掉网页导航、版权栏、重复段落等噪声。
4. 把资料转成统一 Markdown 小节，便于 RAG 系统切分和检索。
5. 生成采集日志和清洗统计，证明处理过程可复查。
6. 最终正式问答仍使用 `knowledge/` 中经过整理的资料，并在文件末尾记录整理人员 Kevin 和整理时间。

这样设计的好处是：资料来源清楚、清洗规则明确、初稿和正式知识库分离，既方便展示成员5工作量，也不会影响系统正式问答效果。
