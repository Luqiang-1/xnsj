# 小组分工建议

项目主题：果用经济作物知识库问答系统设计与实现

本项目采用“数据采集 + 本地知识库 + 检索问答 + Web 页面展示”的方式实现。五名成员的分工建议围绕一条完整流程展开：成员 5 负责资料获取和入库工具，成员 2 负责知识内容整理和校对，成员 4 负责检索与答案质量，成员 1 负责后端框架和大模型预留接口，成员 3 负责前端页面和交互展示。

## 成员 1：系统框架、后端接口与大模型预留

负责文件：`app.py`、`config.py`、`rag_engine.py`

具体任务：

- 维护本地 Web 服务入口，保证运行 `python app.py` 后能正常访问问答页面。
- 维护 `/api/ask` 问答接口，接收前端问题，并返回答案、引用来源和检索片段。
- 维护 `/api/health` 健康检查接口，用于查看系统是否正常启动、知识库加载了多少片段。
- 维护 `/api/reload` 重新加载接口，方便新增或修改知识库后快速更新。
- 处理空问题、非法 JSON、接口路径错误等异常情况，保证系统运行稳定。
- 配置 `config.py` 中的端口、知识库路径、最大返回片段数、最低匹配分数等参数。
- 把“检索知识片段”和“生成最终答案”分成相对清晰的流程，为后续接入大模型预留位置。
- 设计大模型扩展方案，例如后续可以把检索到的知识片段交给本地大模型或云端大模型进行答案改写。

需要写进报告的内容：

- 系统总体架构和后端运行流程。
- 后端接口设计，包括接口地址、请求方式、输入输出字段。
- `app.py` 如何调用 `rag_engine.py` 完成问答。
- 后续接入大模型的设计思路：先检索知识库，再由大模型基于证据生成自然语言答案。

最终交付：

- 系统可以正常启动和访问。
- 前端可以通过接口完成问答。
- 修改知识库后可以重新加载。
- 后端结构清楚，方便后续接入大模型或向量数据库。

## 成员 2：知识内容整理与人工校对

负责文件：`knowledge/草莓.md`、`knowledge/葡萄.md`，以及其他经过整理后的知识库 Markdown 文件

具体任务：

- 负责知识库内容质量，不负责大量网页抓取，主要做筛选、整理、补充和人工校对。
- 根据成员 5 采集或整理出的原始资料，判断内容是否适合放入知识库。
- 按统一 Markdown 结构整理作物资料，建议包含：
  - 品种介绍
  - 种植方式
  - 环境需求
  - 常见病虫害与解决方案
  - 农药使用
  - 采收与销售
- 对草莓、葡萄等重点作物进行重点完善，保证演示问题都能从知识库中找到依据。
- 检查关键数据是否合理，例如温度、湿度、光照、土壤、采收时间、农药安全间隔期等。
- 把大段资料改写成适合问答系统引用的短段落，避免内容过长、过散或只有标题没有正文。
- 统一小节标题和表达方式，让成员 4 的检索规则更容易命中。
- 在资料中保留必要的来源说明，方便展示知识库可靠性。

需要写进报告的内容：

- 知识库内容建设方法。
- 作物资料模板。
- 已整理的作物种类和内容覆盖范围。
- 人工校对和资料可靠性说明。

最终交付：

- 至少完善 2 种重点作物资料。
- 每份资料结构统一、内容清楚、适合检索。
- 常见演示问题能在知识库中找到明确依据。
- 知识库内容经过人工检查，不只是直接复制或直接由大模型生成。

## 成员 3：前端页面与用户交互

负责文件：`static/index.html`、`static/styles.css`、`static/app.js`

具体任务：

- 维护问答页面结构，包括标题区、问题输入框、提交按钮、答案区域和来源区域。
- 在 `static/app.js` 中实现用户输入问题、点击提交、调用 `/api/ask`、显示答案的完整流程。
- 增加加载状态，例如提交问题后显示“正在检索知识库”，提升用户体验。
- 处理异常提示，例如问题为空、服务未启动、接口请求失败等情况。
- 优化答案展示格式，让编号、作物名称、小节名称和正文更容易阅读。
- 优化来源展示，清楚显示资料标题、章节、文件路径和匹配分数。
- 增加推荐问题按钮，方便课堂演示时快速触发典型问答。
- 优化移动端和不同屏幕尺寸下的显示效果。
- 若后续增加知识库管理功能，可配合成员 5 设计简单的“重新加载知识库”或“查看知识库状态”入口。

需要写进报告的内容：

- 前端页面功能说明。
- 用户操作流程说明。
- 页面截图，包括首页、提问中状态、回答结果和来源展示。

最终交付：

- 用户能在页面上顺利完成问答。
- 页面能展示答案和引用来源。
- 推荐问题、加载状态、错误提示等交互完整。
- 页面布局在常见屏幕尺寸下显示正常。

## 成员 4：检索算法、问答质量与防幻觉

负责文件：`rag_engine.py`，可配合修改 `config.py`

具体任务：

- 阅读并解释 `rag_engine.py` 的核心流程：加载 Markdown、切分知识片段、分词、评分、排序、生成答案。
- 优化 `STOPWORDS` 停用词，减少“什么”“如何”“相关”等无意义词对检索结果的影响。
- 优化 `CATEGORY_KEYWORDS` 分类关键词，让系统能识别环境需求、病虫害、农药使用、采收销售等问题类型。
- 调整评分逻辑，例如作物名称命中加分、章节类别命中加分、关键词重叠越多分数越高。
- 调整 `MIN_SCORE`，让系统在证据不足时拒答，而不是强行回答。
- 保证答案只基于检索到的知识片段生成，避免出现知识库没有依据的内容。
- 与成员 2 配合调整知识库小节标题和关键词，让整理后的资料更容易被检索命中。
- 与成员 5 配合检查批量入库的数据是否会影响检索效果，例如标题太乱、内容重复、无关信息太多等。
- 准备正例和反例问题，用来验证系统能正确回答相关问题，也能拒绝无关问题。

需要写进报告的内容：

- 检索算法说明。
- 匹配分数设计。
- 防幻觉策略。
- 正例和反例问题测试结果。

最终交付：

- 常见问题能命中正确作物和正确章节。
- 无关问题能返回“知识库暂无足够依据”。
- 答案中包含明确来源。
- 检索过程有解释性，便于答辩说明。

## 成员 5：数据采集、清洗与知识库入库工具

负责文件：`knowledge/`、`README.md`，后续可新增 `tools/`、`data/` 或数据处理脚本

具体任务：

- 设计知识库数据来源方案，明确哪些资料来自人工整理，哪些资料可以通过公开网页批量采集。
- 负责资料获取流程，可以采用人工收集、复制公开资料、简单爬虫、表格整理等方式，不要求所有数据都手写。
- 如果使用爬虫，只抓取公开、允许访问的农业资料页面，并记录来源链接。
- 对原始资料进行初步清洗，去掉网页导航、广告、重复段落、无关介绍等内容。
- 将清洗后的资料转换成适合成员 2 校对的 Markdown 初稿。
- 设计资料来源记录格式，例如来源网站、来源链接、采集时间、整理人员。
- 设计知识库入库流程：采集资料、清洗内容、转换 Markdown、人工校对、放入 `knowledge/`、重新加载系统。
- 在 `README.md` 中补充“如何新增知识库资料”的操作说明。
- 可规划后续知识库管理功能，例如上传 Markdown、查看已加载文件、点击重新加载知识库等。

需要写进报告的内容：

- 数据来源方案。
- 资料采集或爬虫流程。
- 数据清洗方法。
- Markdown 转换和入库流程。
- 知识库管理功能的后续扩展设计。

最终交付：

- 一套清楚的数据采集和入库流程。
- 若时间允许，提供简单的数据处理脚本或示例转换文件。
- 交给成员 2 的 Markdown 初稿结构基本统一。
- 知识库资料中保留来源说明。
- `README.md` 中包含新增知识库资料的步骤。

## 推荐测试问题

正例问题：

- 草莓适合什么环境？
- 草莓灰霉病怎么防治？
- 草莓农药使用有哪些注意事项？
- 葡萄有哪些常见病虫害？
- 阳光玫瑰葡萄销售时要注意什么？
- 葡萄采收和运输要注意什么？

反例问题：

- 水稻怎么插秧？
- 玉米如何追肥？
- 苹果手机怎么维修？
- 今天上海天气怎么样？
- 帮我写一首诗。

## 小组协作流程

- 成员 5 先确定数据来源，整理出可入库的 Markdown 初稿。
- 成员 2 对 Markdown 初稿进行人工校对、补充和规范化。
- 成员 4 根据整理好的知识库调整检索规则和拒答阈值。
- 成员 1 保证后端接口稳定，并为大模型接入保留扩展空间。
- 成员 3 负责把问答结果、引用来源和系统状态清楚展示给用户。
- 每次修改知识库后，使用 `http://127.0.0.1:8000/api/reload` 重新加载，再进行问答测试。
- 最终答辩时可以按“项目背景、数据来源、知识库建设、系统架构、检索算法、页面展示、测试结果、总结展望”的顺序介绍。
